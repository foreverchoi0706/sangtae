# 과제 테스트 안내

## 과제 테스트에 관하여

1. 과제 테스트는 기술 면접 전, 지원자의 개발 역량 (문제 정의 및 기술적 해결 능력)을 살펴보기 위한 과정입니다. 따라서 문제를 풀이하는데 있어 어떤 도구를 사용하거나, 문서를 참조하는데 있어 제약을 두지 않습니다. 당사는 지원자가 **AI 도구를 포함한 다양한 툴을 적극적으로 사용하는 것을 권장합니다.**

2. 모든 지원자는 주어진 문제들에 대해 문제가 요구하는 코드와 문서를 포함한 산출물들을 안내되는 방식에 따라 기한 내에 제출하여야 합니다.

   1. 제약 조건 내에서 요구되는 기능을 구현하는 것만이 목표가 아니며, 기능적인 완성도 외에도 문서 및 코드의 품질도 중요한 평가 기준이 될 수 있습니다.

3. 일반적인 라이브 코딩 테스트와 달리 과제 테스트에서는 질문자에게 자유롭게 질문하고 대답을 확인한 후 문제 풀이를 진행하기 어렵습니다. 꼭 확인하고자하는 사항이 있는 경우 recurit_tech@medistream.co.kr 로 문의 주세요.

4. 지원자가 제출한 모든 코드 및 문서는 채용 목적으로만 사용되며, 다른 지원자에게 공유되거나 예시로 제출되지 않습니다. 또한 지원자는 어떤 방식으로든 본인에게 제시되었던 과제의 일부 또는 전부를 다른 사람에게 알려주거나, 공유 또는 공개할 수 없습니다.

5. 과제 테스트에서 제출된 내용은 기술 면접시 질문 및 토론을 위한 자료로 사용될 수 있습니다.

## 과제 결과물 구성 - 공통

1. 문제에서 제출할 것을 요구하는 모든 문서는 Github-flavored Markdown Format으로 작성하여야 하며, 반드시 **README.md** 파일이 최상위 디렉토리에 있어야 합니다.

2. **README.md** 에는 아래와 같은 요소들이 포함되어야 합니다.

   1. 지원자 정보
      1. 지원자 이름
      2. 지원 직무
   2. 과제 정보
      1. 과제 코드 - 각 문제별로 주어지는 코드이며 문제 파일 제목 또는 내용에 포함되어있습니다
      2. 과제 수령일
      3. 소요 시간 - 과제 수령 ~ 답안 작성 완료까지 실제로 사용한 시간으로, (휴식 및 다른 개인적인 사유로 소비된 시간을 제외하고) 전체적으로 몇 시간을 사용했는지를 대략적으로 기록합니다.
      4. 수행 환경 - 과제를 수행하는데 사용한 여러 도구들 (IDE, AI Agent 및 기타)과, 참조한 문서/사이트 등을 기술합니다.
      5. 결과 요약 - 과제에서 제시된 기능들 및 제약 조건들 중 제출물에서 구현이 완료된 것, 부분적으로만 완료된 것, (선택 사항으로서) 손대지 않은 것들을 구분하여 명시합니다.
      6. 수행중 가정한 사항들 - 문제의 요구 조건이 불명확하거나 자유롭게 선택 가능한 경우, 특정한 조건을 임의로 가정하여 설계-구현했다면 무엇을 어떻게 가정하였는지 명시합니다.
   3. 과제 산출물 정보
      1. 디렉토리/파일 구조 - 소스 코드, 문서 파일등을 포함하여 산출물의 전체적인 디렉토리 구조를 명시합니다.
      2. 빌드 및 실행 방법 - 제출된 코드가 요구 조건을 만족시키는지 확인하기 위한 빌드 및 실행 과정을 설명합니다. 필수적인 과정들이 빠짐 없이 기술되어야 하며, 평가자가 copy & paste 로 따라 했을 때 문제 없이 실행되어야 합니다.
      3. 기타 의존성 - 과제 문제 해결을 위해 추가적으로 도입한 library로서 문제의 boiler plate 또는 기술 제약 조건에 명시되지 않은 것들은 무엇을 왜 사용하는지 명시합니다. (직접 의존성을 갖는 패키지/도구들만 명시해도 됩니다)
   4. 기타 과제 피드백
      1. 과제 난이도 - 문제 난이도 및 시간이 적절하였는지 지원자 입장에서 평가해주시면 감사하겠습니다.
      1. 제출은 하지 않았지만 - 시간상 구현하지 못했거나, 시간이 있었다면 해낼 수 있었을 것 같은 기능들이 있다면 뭘 어떻게 하려고 했었는지 간단히 알려주세요.
      1. 필수/선택 요구 사항에는 없으나 추가로 만들어본 부분들 - 어떤 기능들인지, 어떤 것들인지 간단히 기술해주세요.

3. 과제가 특정 도구 (library, framework 및 기타) 을 사용할 것을 요구하거나, boiler plate code가 dependency로서 특정 도구를 포함한 경우, 해당 도구를 이용하여 답안을 작성하여야 합니다. 요구 조건에 특정한 버전이 지정되지 않았다면 해당 도구의 최신 안정 버전을 사용해주세요.

4. 문서에 그림/이미지를 포함시키고자할 경우, .png, .svg 등 일반적인 그래픽 포맷 파일을 이용하거나, mermaid 등의 도구를 통해 문서 내에 직접 포함시켜야 합니다.

5. 과제 결과물 제출시, .zip 등 일반적인 압축 포맷 파일을 이용해야 합니다.

6. 제출된 과제 결과물은 반드시 실제 동작과 결과를 확인할 수 있는 방법을 제공해야 하며, 어떻게 실행할 수 있는지 README 파일에 명시해야 합니다.

## 유의 사항 - Backend / Front-end 공통

1. node.js 는 최신 LTS 버전이 시스템에 이미 설치되어있는 상태임을 가정합니다.
2. 과제 수행중 사용한 IDE (vscode, cursor, webstorm등) 없이도 빌드-실행 가능해야 합니다.
3. 실행 전 미리 설치/실행/준비해야 하는 것들은 실행 방법의 일부로서 README 파일에 포함되어야 합니다.
4. (특별한 이유가 없는 한) vanilla javascript 대신 **typescript** 를 사용합니다. 설정, 스크립트등 보조적인 용도로 js를 사용할 수 있습니다.
5. 단순한 기능 구현 외에도, 리소스 관리, 성능/로딩 속도 최적화, 보안 등 구현물의 완성도에도 유의하여야 합니다.
6. 코드의 가독성, 복잡도, 적절한 주석 처리, 일관된 coding convention등 일반적인 코드 품질에도 유의해주세요 .
   1. **eslint** 등 코드 품질 검사 도구를 사용하는 경우 과제 제출시 해당 툴의 설정 파일을 포함시켜야 하며, 빌드 과정에서 해당 툴을 통해 작성된 코드를 실제로 체크하도록 구성해야 합니다.
   2. 해당 도구들은 시스템에 전역으로 설치할 것을 요구하기 보다, project 내부 도구로 설치되도록 구성해주세요
7. AI Coding Assistant 사용을 권장합니다.
   1. **개발중 어떤 AI 도구 (IDE, agent등) 들을 사용하였는지**는 **README** file의 **과제 정보 - 수행 환경**에 포함되어야 합니다.
   2. AI와 협업하기 위해 만든 가이드 및 규칙 문서, 설치한 MCP 서버, 개별 동작을 수행하기 위해 만든 프롬프트 등을 포함한 **Agent의 설정 파일/디렉토리**도 산출물에 포함시켜주세요. (이는 산출물의 동작을 재현하기 위한 것이 아니라 AI와 어떤 방식으로 협업하는지를 보기 위한 목적입니다)

## 유의 사항 - Backend 개발자

1. 제출된 프로그램은 local server 또는 sciript로 실행 가능해야 하며, Windows 및 MacOS에서 local shell을 통해 바로 실행 가능해야 합니다.
   1. 불가피한 경우 docker container image 형태로 빌드하여 docker 호환 container runtime (podman등) 에서 동작시킬 수 있습니다.
   2. node.js 나 docker, docker-compose 등 필수적인 도구들은 이미 설치되어있는 것으로 가정해도 좋습니다.
2. API 서버는 **Open API Specification (OAS, aka Swagger) 규격에 따르는 API 명세**를 제공해야 하며, 모든 request parameter가 API 명세에 포함된 **스키마에 부합하는지 validation**해야 합니다. (예외적인 경우 해당 기능에 왜 validation이 필요하지 않은지 코드 레벨에서 명확히 설명하여야 합니다)
3. API 서버 실행후, 동작을 시연하기 위한 방법이 제공되어야 합니다.
   1. API 서버는 이를 위해 OAS spec 을 api 로 제공하거나, 빌드중 생성하도록 구현하고, 확인 도구로서 [swagger UI](https://swagger.io/tools/swagger-ui/) 를 포함할 수 있습니다.
   2. 간단한 CLI tool이나 web application을 구현하고, 이를 통해 API를 호출하는 등의 동작을 수행하도록 구성할 수 있습니다.
4. express, hapi, koa, fastify등 널리 알려진 **web application framework**에 기반하여야 합니다.
5. 가능한 범위 내에서 jest, vitest등 널리 사용되는 테스트 도구를 이용하여 **unit test**를 구현하고, 테스트 실행후 **coverage report** 를 확인할 수 있어야 합니다.
6. (특별한 이유가 없는 한) API는 **Restful** 하게 설계합니다.
7. (특별한 이유가 없는 한) 모든 데이터를 RDB 및 local file system에 저장하되, DBMS는 **Postgres, MySQL, Sqlite** 등 일반적으로 널리 사용되는 DB를 사용합니다. 단, DBMS는 미리 설치되어있지 않다는 전제 하에서 결과물을 구성하여야 합니다. 특별한 이유가 없다면, DBMS에 종속적이지 않은 형태로 구현되어야 합니다.
8. DB 접근시 Raw SQL 사용을 지양하고 널리 알려진 **ORM 또는 Query builder 를 사용**합니다.
9. 서버에 인입되는 **모든 request 및 response를 console에서 확인 가능하도록** **logging**해야 합니다. 또한, 외부 API를 호출하거나, 이를 mocking 하는 경우 API request 와 response 일체도 logging해야 합니다. 로그는 정확한 시간과, log level (debug, info, warning, error) 을 포함해야 합니다.

## 유의 사항 - Front-end 개발자

1. 제출된 프로그램은 local web server 또는 (webpack, vite 등의 bundler 가 제공하는) 개발 서버에 접근하여 web browser에서 시연 가능해야 합니다.

2. 구현 편의를 위해 [공개된 디자인 시스템들](https://github.com/jbranchaud/awesome-react-design-systems) 을 활용할 수 있습니다. UX/디자인의 완성도는 문제의 요구 사항들을 충족하는 한 중요한 평가 요소로 작용하지 않습니다.

3. 외부 CDN을 통해 공개된 폰트, 아이콘등 디자인 리소스를 사용한 경우, dependency 로서 README 파일에 명시하여야하며, 필요한 경우 app에 content security policy 등의 설정을 미리 포함하여야 합니다.

4. 구현시 외부 dependency (backend-API등) 들은 코드 내에서 직접, 또는 리소스 파일등을 통해 간단히 mocking하거나 가상의 간단한 서버를 포함하는 식으로 구현할 수 있습니다.

5. 프로그램은 웹 브라우저를 통해 동작을 확인 가능해야 하며, 일반적인 개발 환경 (Windows/MacOS의 최신 안정 버전) 에서 보편적으로 사용되는 브라우저(Chrome, Edge, Safari등의 최신 안정 버전)에서 이상 없이 실행 가능해야 합니다.
