# 기술 면접 예상 질문 리스트

## 1. 프로젝트 개요 및 설계 철학

### 1.1 프로젝트 소개

- **Q**: 이 프로젝트를 한 줄로 설명해줘.

  - **A**: Atom 기반의 최소한의 API로 전역 상태, 파생 상태, 비동기 상태를 관리할 수 있는 프레임워크 비종속 상태 관리 라이브러리입니다.

- **Q**: 왜 상태 관리 라이브러리를 직접 구현했어? 기존 라이브러리(Redux, Zustand, Jotai 등)와의 차이점은?

  - **A**: 상태 관리의 핵심 원리를 깊이 이해하고, 실제 프로덕션 수준의 라이브러리를 설계·구현하는 경험을 얻기 위해 직접 구현했습니다. Redux는 Provider 패턴과 액션/리듀서가 필요하지만, 이 라이브러리는 atom-local scope로 구독자를 관리해 Provider 없이도 동작합니다. Zustand는 스토어 단위로 관리하지만, 이 라이브러리는 개별 atom 단위로 세밀한 구독이 가능합니다. Jotai와 유사하지만 프레임워크 비종속 설계로 React/Vue 모두에서 동일한 API를 사용할 수 있습니다.

- **Q**: "Minimal"이라는 키워드가 있는데, 어떤 점에서 최소한이라고 생각해?
  - **A**: 핵심 API가 `createAtom`, `get`, `set`, `subscribe` 네 가지만으로 모든 상태 관리를 할 수 있습니다. Redux처럼 미들웨어나 리듀서 같은 복잡한 개념이 없고, Provider 같은 전역 설정도 필요 없습니다. 또한 프레임워크 바인딩도 단일 `useAtom` 훅 하나로 동기/비동기/파생 atom을 모두 처리할 수 있어 API가 매우 단순합니다.

### 1.2 설계 철학

- **Q**: Atom 기반 설계를 선택한 이유는?

  - **A**: Atom은 상태의 최소 단위로, 각 atom이 독립적으로 값을 보관하고 구독자를 관리합니다. 이렇게 하면 필요한 atom만 구독해 불필요한 리렌더를 방지할 수 있고, 상태를 세밀하게 분리해 관리할 수 있습니다. 또한 atom 단위로 테스트하고 디버깅하기 쉬워집니다.

- **Q**: 왜 프레임워크 비종속(frameworkless)으로 설계했어? 이게 어떤 장점이 있어?

  - **A**: 코어 로직(`@sangtae-js`)을 프레임워크와 분리해 React, Vue, 또는 바닐라 JS 환경에서도 동일한 API로 사용할 수 있습니다. 이렇게 하면 프레임워크를 바꿔도 상태 관리 로직을 재사용할 수 있고, React와 Vue를 혼합 사용하는 프로젝트에서도 동일한 atom을 공유할 수 있습니다. 또한 프레임워크 업그레이드 시 코어 로직에 영향이 없어 유지보수가 쉬워집니다.

- **Q**: Redux의 Provider 패턴 대신 atom-local scope로 구독자를 관리한 이유는?

  - **A**: Provider 패턴은 전역 컨텍스트가 필요하고, 모든 컴포넌트가 Provider로 감싸져야 합니다. atom-local scope는 각 atom이 자체적으로 구독자 Set을 관리하므로 Provider 없이도 동작하고, 특정 atom만 구독하는 컴포넌트만 리렌더됩니다. 이렇게 하면 더 세밀한 성능 제어가 가능하고, atom 단위로 독립적인 테스트가 가능합니다.

- **Q**: "불변성 기반"이라고 했는데, 구체적으로 어떻게 불변성을 유지해?

  - **A**: `set` 함수로 atom 값을 변경할 때, 기존 객체를 직접 수정하지 않고 새로운 값으로 교체합니다. `Object.is` 비교로 값이 실제로 변경되었는지 확인하고, 변경되었을 때만 구독자에게 알립니다. 사용자가 객체를 직접 수정하는 것을 막기 위해 `Atom<T>` 타입을 `Readonly`로 노출해 내부 필드에 직접 접근할 수 없도록 했습니다. 사용자는 항상 새로운 객체를 생성해 `set`으로 전달해야 합니다.

- **Q**: TypeScript 우선 설계라고 했는데, 어떤 점에서 타입 안정성을 강조했어?
  - **A**: 모든 API가 제네릭을 사용해 컴파일 타임에 타입을 보장합니다. `createAtom<T>(initialValue: T)`로 생성하면 이후 `get`, `set`에서 자동으로 타입이 추론됩니다. `useAtom`도 제네릭 없이 atom 타입을 자동 추론해 타입 에러를 사전에 방지합니다. 또한 내부 Symbol 키를 사용해 타입 시스템으로만 접근 가능하도록 설계했습니다.

## 2. 핵심 구현 - Atom과 구독 시스템

### 2.1 Atom 내부 구조

- **Q**: Atom의 내부 구조를 설명해줘. Symbol을 사용한 이유는?

  - **A**: Atom은 내부적으로 `VALUE`와 `LISTENERS` 두 개의 Symbol 키를 가진 객체입니다. `VALUE`는 현재 상태 값을 저장하고, `LISTENERS`는 구독자 콜백들의 Set을 저장합니다. Symbol을 사용한 이유는 외부에서 실수로 접근하거나 수정하는 것을 방지하기 위해서입니다. 일반 문자열 키(`"value"`, `"listeners"`)를 사용하면 외부에서 쉽게 접근할 수 있지만, Symbol은 런타임에 고유하므로 의도하지 않은 접근을 막을 수 있습니다.

- **Q**: `Readonly<InternalAtom<T>>`로 타입을 정의한 이유는? 내부 필드를 외부에서 접근 못하게 막은 이유는?

  - **A**: `Atom<T>` 타입을 `Readonly`로 노출해 외부에서 직접 수정할 수 없도록 했습니다. 이렇게 하면 사용자는 반드시 제공된 API(`get`, `set`, `subscribe`)를 통해서만 상태에 접근해야 하므로, 내부 구조를 보호하고 일관된 상태 관리를 강제할 수 있습니다. 또한 타입 시스템이 컴파일 타임에 잘못된 접근을 차단해 런타임 에러를 방지합니다.

- **Q**: `LISTENERS`를 `Set<Listener<T>> | null`로 관리하는 이유는? null로 초기화하는 이유는?
  - **A**: 구독자가 없을 때는 `null`로 두고, 첫 구독자가 등록될 때만 Set을 생성합니다. 이렇게 하면 구독자가 없는 atom의 메모리 사용량을 최소화할 수 있습니다. 마지막 구독자가 해지될 때도 Set을 `null`로 되돌려 GC가 수거할 수 있도록 합니다. `Set | null` 타입을 사용해 구독자 존재 여부를 명확히 표현하고, null 체크를 통해 안전하게 접근합니다.

### 2.2 값 비교 및 최적화

- **Q**: `Object.is`를 사용한 이유는? `===`와의 차이점은?

  - **A**: `Object.is`는 `===`와 거의 동일하지만 `NaN`과 `+0`/`-0`을 다르게 처리합니다. `Object.is(NaN, NaN)`은 `true`이고 `Object.is(+0, -0)`은 `false`입니다. 상태 관리에서는 `NaN`을 같은 값으로, `+0`과 `-0`을 다른 값으로 구분하는 것이 더 정확한 동작입니다. `===`를 사용하면 `NaN !== NaN`이 되어 불필요한 업데이트가 발생할 수 있습니다.

- **Q**: `NaN`, `+0`, `-0` 같은 엣지 케이스를 어떻게 처리했어?

  - **A**: `Object.is`를 사용해 이러한 엣지 케이스를 정확히 처리합니다. `Object.is(NaN, NaN)`은 `true`이므로 `NaN`으로 설정해도 같은 값으로 인식해 불필요한 업데이트를 방지합니다. `Object.is(+0, -0)`은 `false`이므로 `+0`에서 `-0`으로 변경하면 실제 변경으로 인식해 구독자에게 알립니다. 테스트 코드에서도 이러한 케이스를 명시적으로 검증했습니다.

- **Q**: 객체 참조 비교만으로 충분한가? 깊은 비교(deep equality)가 필요한 경우는 어떻게 해야 할까?
  - **A**: 현재는 `Object.is`로 참조 비교만 합니다. 이는 성능상 이점이 있지만, 내용이 같고 참조가 다른 객체는 다른 값으로 인식됩니다. 깊은 비교가 필요한 경우, 사용자가 `set` 호출 전에 `JSON.stringify`나 `lodash.isEqual` 같은 라이브러리로 비교해 같은 값이면 `set`을 호출하지 않도록 할 수 있습니다. 또는 `createDerivedAtom`에서 깊은 비교를 수행하는 커스텀 atom을 만들 수도 있습니다. 하지만 대부분의 경우 불변성 패턴을 사용하면 참조 비교만으로도 충분합니다.

### 2.3 구독 시스템

- **Q**: `subscribe` 함수의 동작 원리를 설명해줘.

  - **A**: `subscribe`는 atom의 `LISTENERS` Set에 콜백 함수를 추가합니다. 만약 Set이 `null`이면 새로 생성합니다. 이후 `set`으로 값이 변경될 때마다 Set의 모든 콜백을 호출합니다. 함수는 구독 해지 함수를 반환하며, 이 함수를 호출하면 Set에서 해당 콜백을 제거합니다. 마지막 구독자가 해지되면 Set을 `null`로 되돌려 메모리를 정리합니다.

- **Q**: 구독 등록 시 즉시 콜백을 호출하지 않는 이유는? (일부 라이브러리는 등록 시 현재 값을 한 번 호출하는데)

  - **A**: 구독 등록 시 즉시 호출하지 않는 이유는 불필요한 사이드 이펙트를 방지하기 위해서입니다. `useAtom` 같은 훅에서는 초기 렌더 시 `getSnapshot`으로 현재 값을 가져오므로, 구독 시 즉시 호출할 필요가 없습니다. 또한 구독만 하고 값 변경을 기다리는 경우에는 즉시 호출이 오히려 방해가 될 수 있습니다. 필요하다면 사용자가 `subscribe` 후 `get(atom)`을 직접 호출해 현재 값을 얻을 수 있습니다.

- **Q**: `Set`을 사용한 이유는? 배열과 비교했을 때 장단점은?

  - **A**: `Set`을 사용하면 중복 구독을 자동으로 방지하고, 추가/삭제 연산이 O(1)로 빠릅니다. 배열을 사용하면 중복 체크를 위해 `includes`를 호출해야 하고 O(n) 시간이 걸립니다. 또한 Set은 이터러블이므로 `forEach`로 모든 구독자를 순회하기 쉽습니다. 구독자 관리에서는 순서가 중요하지 않고 중복 방지가 중요하므로 Set이 더 적합합니다.

- **Q**: 구독 해지 시 `Set`을 `null`로 되돌리는 이유는? GC와 어떤 관련이 있어?
  - **A**: 마지막 구독자가 해지되면 빈 Set 객체가 남게 됩니다. 빈 Set도 메모리를 차지하므로, `null`로 되돌려 객체 참조를 제거하면 가비지 컬렉터가 수거할 수 있습니다. 이렇게 하면 구독자가 없는 atom의 메모리 사용량을 최소화할 수 있습니다. 나중에 새로운 구독자가 등록되면 다시 Set을 생성하면 되므로, 성능에 큰 영향은 없습니다.

### 2.4 메모리 관리

- **Q**: "GC-friendly"하게 작성했다고 했는데, 구체적으로 어떻게 구현했어?

  - **A**: 두 가지 방식으로 구현했습니다. 첫째, 구독자가 없을 때 `LISTENERS`를 `null`로 설정해 빈 Set 객체를 GC 대상으로 만듭니다. 둘째, 비동기 Atom의 Promise 캐싱에 `WeakMap`을 사용해 Promise가 더 이상 참조되지 않으면 자동으로 캐시가 정리되도록 했습니다. 이렇게 하면 사용하지 않는 리소스가 자동으로 정리되어 메모리 누수를 방지할 수 있습니다.

- **Q**: 마지막 구독자가 해지되면 `LISTENERS`를 `null`로 바꾸는데, 이게 왜 메모리 누수를 방지하는 거야?

  - **A**: JavaScript에서 객체는 참조가 하나라도 남아있으면 GC되지 않습니다. 빈 Set 객체도 메모리를 차지하므로, `null`로 참조를 제거하면 Set 객체에 대한 참조가 없어져 GC가 수거할 수 있습니다. 만약 `null`로 바꾸지 않으면 구독자가 없는 atom도 계속 빈 Set을 유지해 메모리가 누적됩니다. 특히 많은 atom을 생성하고 해지하는 경우에는 이 차이가 중요합니다.

- **Q**: WeakMap을 사용한 부분이 있어? 어디서 사용했고 왜 사용했어?
  - **A**: `createAsyncAtom`에서 Promise를 키로 사용하는 캐시에 `WeakMap`을 사용했습니다. 같은 Promise에 대해 항상 같은 Atom을 반환하기 위해 캐싱이 필요한데, `WeakMap`을 사용하면 Promise가 더 이상 참조되지 않을 때 자동으로 캐시 엔트리가 제거됩니다. 일반 `Map`을 사용하면 Promise 참조를 계속 유지해야 해 메모리 누수가 발생할 수 있습니다. `WeakMap`은 키가 약한 참조이므로 GC가 Promise를 수거하면 캐시도 자동으로 정리됩니다.

## 3. 고급 기능 - 파생 Atom

### 3.1 의존성 추적

- **Q**: `createDerivedAtom`에서 의존성을 어떻게 자동으로 추적해?

  - **A**: `callback` 함수에 `trackedGet`을 전달해, callback 내에서 `get(atom)`을 호출할 때마다 해당 atom을 `nextTrackedAtoms` Set에 추가합니다. 이렇게 하면 callback 실행 중 접근한 모든 atom을 자동으로 추적할 수 있습니다. callback 실행 후 `applySubscriptions`로 새로 추적된 atom들과 기존 atom들을 비교해 구독을 관리합니다.

- **Q**: `trackedGet` 패턴이 뭐야? 어떻게 동작해?

  - **A**: `trackedGet`은 `get` 함수를 래핑한 함수로, `get(atom)`을 호출하기 전에 해당 atom을 의존성 Set에 추가합니다. `trackedGet(atom)`을 호출하면 `nextTrackedAtoms.add(atom)`으로 의존성을 기록하고, 그 다음 `get(atom)`으로 실제 값을 반환합니다. 이렇게 하면 callback 실행 중 어떤 atom을 읽었는지 자동으로 추적할 수 있습니다.

- **Q**: 의존성이 변경되면 어떻게 파생 Atom을 재계산해?
  - **A**: 의존 atom에 `subscribe`를 등록해 값이 변경될 때마다 `evaluate` 함수를 호출합니다. `evaluate`는 `trackedGet`을 사용해 다시 의존성을 추적하고, callback을 실행해 새 값을 계산합니다. 새 값과 이전 값을 `Object.is`로 비교해 다르면 구독자에게 알립니다. 또한 `applySubscriptions`로 의존성이 변경되었는지 확인해, 끊어진 의존성의 구독은 해제하고 새 의존성은 구독합니다.

### 3.2 구독 관리

- **Q**: 파생 Atom의 의존성이 동적으로 변경될 수 있는데, 구독을 어떻게 관리해?

  - **A**: `applySubscriptions` 함수가 매번 재평가 시 새로 추적된 atom Set과 기존 atom Set을 비교합니다. 기존에 있던 atom이 새 Set에 없으면 구독을 해제하고, 새 Set에만 있는 atom이면 새로 구독합니다. 이렇게 하면 조건부로 atom을 읽는 경우에도 동적으로 구독을 관리할 수 있습니다. 예를 들어 `if (condition) get(atomA)` 같은 경우, condition이 false가 되면 atomA 구독이 자동으로 해제됩니다.

- **Q**: `applySubscriptions` 함수가 하는 일은?

  - **A**: `applySubscriptions`는 새로 추적된 atom Set(`nextTrackedAtoms`)을 받아서, 기존 atom Set(`atoms`)과 비교합니다. 기존에 있던 atom이 새 Set에 없으면 `subscriptions` Map에서 해당 atom의 구독 해지 함수를 호출해 구독을 해제합니다. 새 Set에만 있는 atom이면 해당 atom에 `subscribe`를 등록하고 해지 함수를 Map에 저장합니다. 마지막으로 `atoms`를 `nextTrackedAtoms`로 업데이트합니다.

- **Q**: 끊어진 의존성의 구독을 해제하지 않으면 어떤 문제가 생겨?
  - **A**: 끊어진 의존성의 구독을 해제하지 않으면, 더 이상 사용하지 않는 atom의 값이 변경될 때도 불필요하게 파생 atom이 재계산됩니다. 이는 성능 저하로 이어지고, 메모리 누수도 발생할 수 있습니다. 또한 의존성이 동적으로 변경되는 경우(예: 조건부로 atom 읽기)에는 오래된 의존성에 계속 구독되어 있어 잘못된 재계산이 발생할 수 있습니다.

### 3.3 비동기 처리

- **Q**: 파생 Atom에서 Promise를 throw하는 이유는? Suspense와 어떻게 연동해?

  - **A**: 파생 atom의 callback에서 비동기 atom을 읽을 때, 그 atom이 아직 완료되지 않았으면 Promise를 throw합니다. React Suspense는 컴포넌트 렌더링 중 Promise가 throw되면 해당 컴포넌트를 일시 중단하고, Promise가 완료되면 다시 렌더링합니다. Proxy의 `get` 트랩에서 `VALUE` 접근 시 `PENDING` 상태면 `state.suspense` Promise를 throw해 Suspense 플로우를 유지합니다.

- **Q**: `state.suspense === thrown` 비교가 왜 필요해? Race condition을 방지하기 위한 거야?

  - **A**: 네, 맞습니다. 여러 비동기 작업이 동시에 실행되면, 나중에 시작된 Promise가 먼저 완료될 수 있습니다. `state.suspense === thrown` 비교를 통해 현재 추적 중인 Promise가 맞는지 확인하고, 다른 Promise가 완료되었을 때는 상태를 업데이트하지 않습니다. 이렇게 하면 오래된 Promise의 결과로 상태가 덮어쓰이는 것을 방지할 수 있습니다.

- **Q**: Promise가 reject되면 어떻게 처리해?
  - **A**: Promise의 `catch` 핸들러에서 `state.status`를 `ERROR`로 설정하고 `state.error`에 에러를 저장한 다음 `emit()`으로 구독자에게 알립니다. Proxy의 `get` 트랩에서 `ERROR` 상태면 `state.error`를 throw해 사용자가 try-catch로 처리할 수 있도록 합니다. `suppressErrors`가 false면 `evaluate` 함수에서도 에러를 다시 throw해 상위로 전파합니다.

### 3.4 Proxy 사용

- **Q**: 파생 Atom에서 Proxy를 사용한 이유는?

  - **A**: 파생 atom은 `PENDING`, `ERROR` 상태일 때 `get` 호출 시 Promise나 에러를 throw해야 합니다. 일반 객체에서는 `get(atom)`이 단순히 값을 반환하지만, Proxy를 사용하면 `get` 트랩에서 상태를 확인하고 필요시 Promise나 에러를 throw할 수 있습니다. 또한 `set` 트랩에서도 값 비교와 상태 업데이트를 수행할 수 있어 Suspense 패턴을 자연스럽게 구현할 수 있습니다.

- **Q**: Proxy의 `get` 트랩에서 `VALUE` 접근만 가로채는 이유는?

  - **A**: `VALUE`는 실제 값이 저장된 Symbol 키이므로, `get(atom)` 호출 시 내부적으로 `atom[VALUE]`에 접근합니다. `LISTENERS` 같은 다른 속성은 가로채지 않아야 정상 동작합니다. `property === VALUE`일 때만 상태를 확인하고 Promise/에러를 throw하거나 값을 반환하면, 다른 속성 접근은 `Reflect.get`으로 원래 동작을 유지할 수 있습니다.

- **Q**: Proxy를 사용하지 않고 다른 방법으로 구현할 수 있을까?
  - **A**: 가능하지만 더 복잡해집니다. 예를 들어 `get` 함수에서 atom 타입을 확인해 파생 atom이면 상태를 확인하고 throw하는 방법이 있지만, 이는 모든 `get` 호출에서 타입 체크가 필요해 성능이 떨어집니다. 또는 파생 atom을 완전히 다른 구조로 만들 수도 있지만, 이는 일반 atom과의 일관성을 해칩니다. Proxy를 사용하면 일반 atom과 동일한 인터페이스를 유지하면서도 특수한 동작을 구현할 수 있어 가장 깔끔한 방법입니다.

## 4. 고급 기능 - 비동기 Atom

### 4.1 캐싱 전략

- **Q**: `WeakMap`을 사용해서 Promise를 캐싱하는 이유는?

  - **A**: 같은 Promise에 대해 항상 같은 Atom을 반환하기 위해 캐싱이 필요합니다. `WeakMap`을 사용하면 Promise가 더 이상 참조되지 않을 때 자동으로 캐시 엔트리가 제거되어 메모리 누수를 방지할 수 있습니다. 일반 `Map`을 사용하면 Promise 참조를 계속 유지해야 해 메모리 누수가 발생할 수 있습니다.

- **Q**: 같은 Promise에 대해 같은 Atom을 반환하는 게 왜 중요해?

  - **A**: 같은 API 요청을 여러 컴포넌트에서 사용할 때, 각각 다른 Atom을 생성하면 중복 요청이 발생하고 상태가 동기화되지 않습니다. 같은 Promise에 대해 같은 Atom을 반환하면 여러 컴포넌트가 동일한 Atom을 구독해 상태를 공유할 수 있습니다.

- **Q**: WeakMap을 사용하지 않고 일반 Map을 사용하면 어떤 문제가 생겨?
  - **A**: 일반 Map을 사용하면 Promise 객체를 키로 유지해야 하므로, Promise가 완료되어도 Map에 참조가 남아있어 GC되지 않습니다. 이는 메모리 누수로 이어집니다. WeakMap은 약한 참조를 사용하므로 Promise가 더 이상 사용되지 않으면 자동으로 캐시에서 제거됩니다.

### 4.2 Suspense 연동

- **Q**: `createAsyncAtom`에서 Promise를 throw하는 이유는?

  - **A**: React Suspense 패턴과 호환되기 위해서입니다. Suspense는 컴포넌트 렌더링 중 Promise가 throw되면 해당 컴포넌트를 일시 중단하고, Promise가 완료되면 다시 렌더링합니다. `get` 호출 시 Promise를 throw하면 Suspense가 이를 감지해 자동으로 로딩 상태를 처리할 수 있습니다.

- **Q**: React Suspense와 어떻게 연동되는지 설명해줘.

  - **A**: `get(asyncAtom)` 호출 시 Promise가 아직 완료되지 않았으면 Proxy의 `get` 트랩에서 Promise를 throw합니다. React 컴포넌트에서 `useAtom(asyncAtom)`을 사용하면 `getSnapshot`에서 Promise가 throw되고, React Suspense가 이를 감지해 fallback UI를 표시합니다. Promise가 완료되면 자동으로 다시 렌더링되어 결과를 표시합니다.

- **Q**: `getAsync` 헬퍼 함수를 추가한 이유는? Suspense가 없는 환경에서 어떻게 사용해?

  - **A**: Suspense가 없는 환경(바닐라 JS, Vue 등)에서도 비동기 Atom을 쉽게 사용할 수 있도록 추가했습니다. `getAsync`는 내부적으로 `get`을 호출하고, Promise가 throw되면 `await`로 기다려 값을 반환합니다. 이렇게 하면 try-catch 없이 async/await로 사용할 수 있어 코드가 간결해집니다.

- **Q**: `getAsync`를 사용하면 try-catch 없이 async/await로 사용할 수 있다고 했는데, 내부적으로 어떻게 동작해?
  - **A**: `getAsync`는 `get(atom)`을 try-catch로 감싸서 호출합니다. Promise가 throw되면 catch 블록에서 이를 감지하고 `await thrown`으로 Promise를 기다립니다. Promise가 완료되면 값을 반환하고, 에러가 발생하면 다시 throw합니다. 이렇게 하면 사용자는 `const value = await getAsync(atom)`처럼 간단하게 사용할 수 있습니다.

### 4.3 상태 관리

- **Q**: 비동기 Atom의 상태를 `pending/success/error`로 관리하는 이유는?

  - **A**: 세 가지 상태를 명확히 구분해 각 상태에 맞는 동작을 수행하기 위해서입니다. `pending`일 때는 Promise를 throw하고, `success`일 때는 값을 반환하며, `error`일 때는 에러를 throw합니다. 이렇게 하면 사용자가 각 상태를 명확히 처리할 수 있습니다.

- **Q**: `set`으로 비동기 Atom의 값을 직접 설정할 수 있는데, 이게 왜 필요해?

  - **A**: 비동기 요청이 완료되기 전에 사용자가 값을 직접 설정하고 싶을 수 있습니다. 예를 들어 낙관적 업데이트(optimistic update)를 구현하거나, 캐시된 값을 미리 설정할 때 유용합니다. `set`을 호출하면 상태가 `success`로 변경되어 이후 `get` 호출 시 즉시 값을 반환할 수 있습니다.

- **Q**: Promise가 완료되기 전에 `set`을 호출하면 어떻게 동작해?
  - **A**: `set`을 호출하면 Proxy의 `set` 트랩에서 값을 설정하고 상태를 `success`로 변경합니다. 이후 Promise가 완료되어도 이미 `success` 상태이므로 Promise의 결과는 무시됩니다. 하지만 Promise의 `then` 핸들러는 여전히 실행되므로, 필요하다면 Promise 완료 후에도 값을 업데이트할 수 있습니다.

## 5. 프레임워크 연동 - React

### 5.1 useSyncExternalStore

- **Q**: `useAtom`에서 `useSyncExternalStore`를 사용한 이유는?

  - **A**: `useSyncExternalStore`는 React 18에서 외부 스토어를 구독하는 공식 훅입니다. 이 훅을 사용하면 Concurrent Rendering에서도 안전하게 동작하고, 서버 사이드 렌더링과도 호환됩니다. 또한 React가 자동으로 배치 업데이트를 처리해주므로 별도의 배치 로직이 필요 없습니다.

- **Q**: `useSyncExternalStore`의 세 번째 인자로 `getSnapshot`을 또 전달한 이유는?

  - **A**: 세 번째 인자는 서버 사이드 렌더링(SSR) 시 사용되는 서버 스냅샷 함수입니다. SSR 환경에서는 클라이언트와 서버의 스냅샷이 일치해야 하므로, 서버에서도 동일한 `getSnapshot`을 사용해 일관성을 보장합니다. 클라이언트에서는 두 번째 인자만 사용되지만, 타입 일관성을 위해 같은 함수를 전달합니다.

- **Q**: `useSyncExternalStore` 대신 `useState` + `useEffect`를 사용하면 안 돼?

  - **A**: 가능하지만 문제가 있습니다. `useState` + `useEffect`는 Concurrent Rendering에서 스냅샷 불일치 문제가 발생할 수 있고, SSR과 호환되지 않습니다. 또한 React 18의 자동 배치와 제대로 연동되지 않아 불필요한 리렌더가 발생할 수 있습니다. `useSyncExternalStore`는 이러한 문제들을 모두 해결해주는 공식 훅입니다.

- **Q**: Concurrent Rendering에서 안전하게 동작한다고 했는데, 어떻게 보장해?

  - **A**: `useSyncExternalStore`는 내부적으로 스냅샷을 비교해 변경이 있을 때만 리렌더를 트리거합니다. 또한 렌더링 중에 스냅샷이 변경되면 경고를 발생시켜 일관성을 보장합니다. 이렇게 하면 Concurrent Rendering 중에 상태가 변경되어도 안전하게 처리할 수 있습니다.

- **Q**: React 18+ 환경이 필요한 이유는?
  - **A**: `useSyncExternalStore`는 React 18에서 추가된 훅입니다. React 17 이하에서는 이 훅이 없어 사용할 수 없습니다. 또한 React 18의 자동 배치 기능과 연동되어 더 효율적으로 동작합니다.

### 5.2 메모이제이션

- **Q**: `useCallback`을 사용해서 구독/스냅샷 함수를 메모이제이션한 이유는?

  - **A**: `useSyncExternalStore`의 첫 번째와 두 번째 인자는 매 렌더마다 새로 생성되면 구독/해제가 반복됩니다. `useCallback`으로 메모이제이션하면 atom이 변경되지 않는 한 같은 함수 참조를 유지해 불필요한 구독/해제를 방지할 수 있습니다.

- **Q**: `useCallback`의 의존성 배열에 `atom`만 있는데, 이게 안전해?

  - **A**: 네, 안전합니다. `subscribe`와 `get` 함수는 atom을 클로저로 캡처하므로, atom이 변경되지 않는 한 항상 최신 atom을 참조합니다. `atom`이 변경되면 새로운 함수가 생성되어 구독도 자동으로 업데이트됩니다.

- **Q**: 메모이제이션을 안 하면 어떤 문제가 생겨?
  - **A**: 매 렌더마다 새로운 구독 함수가 생성되어 `useSyncExternalStore`가 구독을 해제하고 다시 등록합니다. 이는 불필요한 작업이고, 구독 해제/등록 사이에 값 변경이 발생하면 업데이트를 놓칠 수 있습니다. 또한 성능 저하도 발생합니다.

### 5.3 배치 업데이트

- **Q**: 배치 업데이트를 어떻게 구현했어?

  - **A**: React 18의 자동 배치 기능을 활용했습니다. `useSyncExternalStore`는 내부적으로 React의 배치 메커니즘과 연동되어, 같은 이벤트 핸들러나 effect 내에서 여러 atom을 업데이트해도 한 번만 리렌더됩니다. 별도의 배치 API를 구현할 필요 없이 React가 자동으로 처리합니다.

- **Q**: React 18의 자동 배치와 어떻게 연동되는지 설명해줘.

  - **A**: React 18에서는 이벤트 핸들러, effect, timeout 등 모든 곳에서 자동으로 배치됩니다. `set(atomA, 1); set(atomB, 2)`처럼 연속으로 호출하면, 각 `set`이 구독자에게 알리지만 `useSyncExternalStore`는 배치 내에서 한 번만 리렌더를 스케줄링합니다. 이렇게 하면 여러 atom 업데이트가 한 번의 리렌더로 처리됩니다.

- **Q**: 별도의 배치 API 없이 React API와 연동해서 구현했다고 했는데, 구체적으로 어떻게 했어?
  - **A**: `useSyncExternalStore`가 내부적으로 React의 배치 메커니즘을 사용합니다. `set` 호출 시 구독자에게 즉시 알리지만, `useSyncExternalStore`의 `onStoreChange` 콜백은 React의 배치 큐에 들어갑니다. React가 배치를 플러시할 때 한 번만 리렌더를 수행하므로, 별도의 배치 API 없이도 자동으로 배치됩니다.

## 6. 프레임워크 연동 - Vue

### 6.1 Composition API

- **Q**: Vue에서 `ref`를 사용해서 Atom 값을 래핑한 이유는?

  - **A**: Vue의 반응성 시스템은 `ref`를 통해 동작합니다. `ref`로 래핑하면 Vue가 값 변경을 감지해 템플릿을 자동으로 업데이트합니다. Atom 값이 변경되면 `ref.value`를 업데이트하고, Vue가 이를 감지해 관련 컴포넌트를 리렌더링합니다.

- **Q**: React와 Vue 연동 방식의 차이점은?

  - **A**: React는 `useSyncExternalStore`로 외부 스토어를 구독하고, Vue는 `ref` + `subscribe`로 반응성을 구현합니다. React는 훅이 값을 직접 반환하지만, Vue는 `Ref<T>`를 반환해 템플릿에서 자동 언래핑됩니다. 또한 Vue는 `onUnmounted`로 구독을 해제하지만, React는 `useSyncExternalStore`가 자동으로 처리합니다.

- **Q**: Vue의 반응성 시스템과 Atom을 어떻게 통합했어?

  - **A**: Atom의 `subscribe`로 값 변경을 감지하고, 변경 시 `ref.value`를 업데이트합니다. Vue는 `ref.value` 변경을 감지해 자동으로 템플릿을 업데이트합니다. 이렇게 하면 Atom의 구독 시스템과 Vue의 반응성 시스템이 자연스럽게 연동됩니다.

- **Q**: Vue에서 `useAtom`이 `Ref<T>`를 반환하는 이유는? 템플릿에서 자동 언래핑이 되는 이유는?

  - **A**: Vue 3의 템플릿은 `ref`를 자동으로 언래핑합니다. `<div>{{ count }}</div>`처럼 사용하면 `count.value`로 자동 접근됩니다. `Ref<T>`를 반환하면 템플릿에서 `.value` 없이 사용할 수 있어 더 자연스럽습니다. 스크립트에서는 `count.value`로 접근해야 합니다.

- **Q**: Vue 3 환경이 필요한 이유는?
  - **A**: Composition API와 `ref`, `onUnmounted` 같은 기능이 Vue 3에서 도입되었습니다. Vue 2에서는 Options API만 있어 Composition API 기반의 `useAtom`을 구현하기 어렵습니다. 또한 Vue 3의 반응성 시스템이 더 효율적이고 타입스크립트 지원이 좋습니다.

### 6.2 라이프사이클 관리

- **Q**: `onUnmounted`에서 구독을 해제하는 이유는?

  - **A**: 컴포넌트가 언마운트되면 더 이상 해당 컴포넌트의 구독이 필요 없습니다. 구독을 해제하지 않으면 메모리 누수가 발생하고, 언마운트된 컴포넌트의 콜백이 계속 실행될 수 있습니다. `onUnmounted`는 컴포넌트 생명주기의 정리 단계이므로 여기서 구독을 해제하는 것이 적절합니다.

- **Q**: React의 `useEffect`와 Vue의 `onUnmounted`의 차이점은?
  - **A**: `useEffect`는 cleanup 함수를 반환해 정리 작업을 수행하지만, `onUnmounted`는 별도의 훅으로 명시적으로 호출합니다. `useEffect`는 의존성 배열로 실행 시점을 제어하지만, `onUnmounted`는 언마운트 시에만 실행됩니다. Vue에서는 구독 해제를 `onUnmounted`에서 명시적으로 처리하는 것이 더 명확합니다.

### 6.3 Suspense 처리

- **Q**: Vue에서 비동기 Atom의 Promise를 어떻게 처리해?

  - **A**: `get(atom)` 호출 시 Promise가 throw될 수 있으므로 try-catch로 감쌉니다. Promise가 throw되면 `promise.then()`으로 완료를 기다린 후 구독을 시작합니다. 초기값은 `undefined`로 설정하고, Promise가 완료되면 `ref.value`를 업데이트합니다. Vue의 `<Suspense>` 컴포넌트와도 연동할 수 있습니다.

- **Q**: React Suspense와 Vue Suspense의 차이점은?
  - **A**: React Suspense는 Promise를 throw하는 방식으로 동작하지만, Vue Suspense는 `async setup`이나 Promise를 반환하는 컴포넌트를 감싸는 방식입니다. React는 렌더링 중 Promise throw를 감지하지만, Vue는 컴포넌트의 비동기 상태를 명시적으로 처리합니다. 하지만 둘 다 비동기 작업 완료를 기다리는 목적은 동일합니다.

## 7. 성능 최적화

### 7.1 렌더링 최적화

- **Q**: `Object.is` 비교로 불필요한 렌더를 어떻게 막았어?

  - **A**: `set` 호출 시 `Object.is`로 새 값과 이전 값을 비교합니다. 같으면 구독자에게 알리지 않아 리렌더가 발생하지 않습니다. 파생 atom도 재계산 후 `Object.is`로 비교해 값이 같으면 구독자에게 알리지 않습니다. 이렇게 하면 불필요한 리렌더를 방지할 수 있습니다.

- **Q**: 파생 Atom의 값이 같으면 구독자에게 알리지 않는 이유는?

  - **A**: 값이 실제로 변경되지 않았는데 구독자에게 알리면 불필요한 리렌더가 발생합니다. 예를 들어 `get(atomA) + get(atomB)`에서 atomA가 변경되었지만 결과가 같으면(예: 1+2=3, 2+1=3) 알릴 필요가 없습니다. `Object.is`로 비교해 실제로 값이 변경되었을 때만 알립니다.

- **Q**: 어떤 경우에 불필요한 리렌더가 발생할 수 있어?
  - **A**: 객체나 배열을 새로 생성해 `set`에 전달하면 참조가 달라서 항상 변경으로 인식됩니다. 예를 들어 `set(atom, { ...get(atom) })`처럼 스프레드 연산자를 사용하면 내용이 같아도 새 객체이므로 리렌더가 발생합니다. 이를 방지하려면 실제로 값이 변경되었을 때만 새 객체를 생성해야 합니다.

### 7.2 구독 최적화

- **Q**: `Set`을 사용한 구독자 관리의 시간 복잡도는?

  - **A**: `Set`의 `add`, `delete`, `has` 연산은 모두 O(1)입니다. 구독자 추가/삭제가 O(1)이므로 구독자가 많아도 성능이 유지됩니다. `forEach`로 모든 구독자를 순회하는 것은 O(n)이지만, 이는 구독자 수에 비례하므로 정상적인 비용입니다.

- **Q**: 의존성 추적에서 `Set`을 사용한 이유는?

  - **A**: 의존성 atom들을 Set으로 관리하면 중복을 자동으로 방지하고, `has`로 빠르게 확인할 수 있습니다. `applySubscriptions`에서 기존 atom과 새 atom을 비교할 때 `Set.has`로 O(1) 시간에 확인할 수 있어 효율적입니다.

- **Q**: 중복 구독을 어떻게 방지해?
  - **A**: `Set`은 같은 참조를 중복 저장하지 않습니다. 같은 콜백 함수를 여러 번 `subscribe`에 전달해도 Set에 한 번만 저장되어 중복 호출이 발생하지 않습니다. 하지만 다른 함수 참조를 전달하면 각각 별도로 저장됩니다.

### 7.3 메모리 최적화

- **Q**: WeakMap을 사용한 캐싱이 메모리 측면에서 어떤 이점이 있어?

  - **A**: Promise가 더 이상 참조되지 않으면 자동으로 캐시에서 제거되어 메모리가 해제됩니다. 일반 Map을 사용하면 Promise 참조를 계속 유지해야 해 메모리 누수가 발생할 수 있습니다. WeakMap은 약한 참조를 사용하므로 GC가 Promise를 수거하면 캐시도 자동으로 정리됩니다.

- **Q**: 구독자 Set을 null로 되돌리는 게 실제로 메모리 사용량을 줄이는지 확인해봤어?
  - **A**: 테스트 코드에서 구독 해제 후에도 atom이 정상 작동하는지 확인했지만, 실제 메모리 사용량 측정은 하지 않았습니다. 하지만 JavaScript의 GC 동작 원리상 참조가 없어지면 객체가 수거되므로, 이론적으로는 메모리 사용량이 줄어듭니다. 프로덕션 환경에서 메모리 프로파일러로 확인하는 것이 좋습니다.

## 8. 테스트 및 검증

### 8.1 테스트 전략

- **Q**: 어떤 테스트 프레임워크를 사용했어? Vitest를 선택한 이유는?

  - **A**: Vitest를 사용했습니다. Vitest는 Vite 기반으로 빠르고, ESM을 네이티브로 지원하며, Jest와 호환되는 API를 제공합니다. 또한 TypeScript 지원이 우수하고 설정이 간단합니다. Vite 프로젝트와 통합하기 쉬워 선택했습니다.

- **Q**: 테스트 커버리지는 어느 정도야?

  - **A**: 필수 기능과 선택 기능을 모두 테스트했습니다. `createAtom`, `get`, `set`, `subscribe`의 기본 동작과 엣지 케이스(NaN, -0 등), 파생 atom의 의존성 추적, 비동기 atom의 Promise 처리, 구독 해제 등 주요 시나리오를 검증했습니다. 하지만 정확한 커버리지 수치는 측정하지 않았습니다.

- **Q**: 엣지 케이스(NaN, -0, Promise reject 등)를 어떻게 테스트했어?
  - **A**: Vitest의 `describe`와 `it`으로 각 엣지 케이스를 별도 테스트 케이스로 작성했습니다. `NaN`과 `-0`은 `Object.is` 비교가 제대로 동작하는지 확인했고, Promise reject는 `createAsyncAtom`에 실패하는 Promise를 전달해 에러가 제대로 throw되는지 확인했습니다. `vi.fn()`으로 콜백 호출 횟수도 검증했습니다.

### 8.2 통합 테스트

- **Q**: React/Vue 샘플 앱을 만든 이유는?

  - **A**: 실제 사용 시나리오를 검증하고, 라이브러리가 프레임워크와 제대로 연동되는지 확인하기 위해서입니다. 또한 사용자가 라이브러리를 쉽게 이해하고 사용할 수 있도록 예제를 제공합니다. 모든 핵심 기능(동기, 파생, 비동기 atom)을 UI로 시연할 수 있습니다.

- **Q**: 실제 사용 시나리오를 어떻게 검증했어?
  - **A**: 샘플 앱에서 카운터, 파생 상태 계산, 비동기 데이터 로딩 등 실제 사용 패턴을 구현했습니다. 컴포넌트별 렌더 횟수를 시각화해 최소 렌더 전략이 제대로 작동하는지 확인했습니다. 또한 React와 Vue 양쪽에서 동일한 시나리오를 테스트해 프레임워크 간 일관성을 검증했습니다.

## 9. 기술적 선택과 트레이드오프

### 9.1 설계 결정

- **Q**: Proxy를 사용한 이유는? 다른 대안은 없었어?

  - **A**: 파생 atom과 비동기 atom에서 `get` 호출 시 상태에 따라 Promise나 에러를 throw해야 합니다. Proxy를 사용하면 일반 atom과 동일한 인터페이스를 유지하면서도 특수한 동작을 구현할 수 있습니다. 다른 방법으로는 `get` 함수에서 타입 체크를 하거나 완전히 다른 구조를 만들 수 있지만, 이는 일관성을 해치거나 성능이 떨어집니다.

- **Q**: TypeScript를 사용한 이유는? 타입 안정성이 어떤 이점을 줘?

  - **A**: 컴파일 타임에 타입 에러를 발견할 수 있어 런타임 에러를 방지합니다. 또한 IDE 자동완성과 리팩토링이 쉬워 개발 생산성이 향상됩니다. 제네릭을 활용해 타입 추론이 잘 되어 사용자가 타입을 명시하지 않아도 됩니다. 라이브러리 사용자에게도 타입 안정성을 제공할 수 있습니다.

- **Q**: 모노레포 구조로 만든 이유는? npm workspace를 선택한 이유는?

  - **A**: 코어 라이브러리(`@sangtae-js`)와 프레임워크 바인딩(`@sangtae-react`, `@sangtae-vue`)을 분리하면서도 함께 관리하기 위해서입니다. npm workspace를 사용하면 각 패키지를 독립적으로 관리하면서도 루트에서 한 번에 빌드하고 테스트할 수 있습니다. 또한 패키지 간 의존성을 쉽게 관리할 수 있습니다.

- **Q**: tsup을 번들러로 선택한 이유는? 다른 번들러(webpack, rollup 등)와 비교했을 때 장점은?

  - **A**: tsup은 esbuild 기반으로 매우 빠르고, TypeScript를 직접 처리하며, ESM/CJS 모두 자동으로 생성합니다. 설정이 간단하고 Zero Config에 가까워 빠르게 시작할 수 있습니다. webpack은 설정이 복잡하고, rollup은 플러그인 설정이 필요하지만 tsup은 기본 설정만으로도 충분합니다.

- **Q**: Vitest를 테스트 프레임워크로 선택한 이유는? Jest와 비교했을 때 장점은?
  - **A**: Vitest는 Vite 기반으로 매우 빠르고, ESM을 네이티브로 지원합니다. Jest와 호환되는 API를 제공해 마이그레이션이 쉽고, TypeScript 지원이 우수합니다. 또한 Vite 프로젝트와 통합하기 쉬워 설정이 간단합니다. Jest는 CommonJS 중심이고 설정이 복잡한 반면, Vitest는 현대적인 도구입니다.

### 9.2 제약사항 및 한계

- **Q**: 현재 구현의 한계점은 뭐야?

  - **A**: 깊은 비교(deep equality)를 지원하지 않아 내용이 같고 참조가 다른 객체는 다른 값으로 인식됩니다. 또한 DevTools나 미들웨어 같은 고급 기능이 없고, 서버 사이드 렌더링을 명시적으로 지원하지 않습니다. 타임트래블이나 undo/redo 같은 기능도 없습니다.

- **Q**: 깊은 비교(deep equality)가 필요한 경우는 어떻게 처리해야 할까?

  - **A**: 사용자가 `set` 호출 전에 `lodash.isEqual` 같은 라이브러리로 비교해 같은 값이면 `set`을 호출하지 않도록 할 수 있습니다. 또는 `createDerivedAtom`에서 깊은 비교를 수행하는 커스텀 atom을 만들 수도 있습니다. 하지만 대부분의 경우 불변성 패턴을 사용하면 참조 비교만으로도 충분합니다.

- **Q**: 서버사이드 렌더링(SSR)을 지원하려면 어떻게 해야 할까?
  - **A**: 서버와 클라이언트에서 각각 독립적인 atom 인스턴스를 생성하고, 서버에서 생성한 상태를 클라이언트로 전달하는 메커니즘이 필요합니다. `useSyncExternalStore`의 세 번째 인자(서버 스냅샷)를 활용하거나, 초기 상태를 하이드레이션하는 API를 추가해야 합니다. 또한 서버에서는 구독이 필요 없으므로 최적화가 필요합니다.

### 9.3 개선 가능한 점

- **Q**: 추가로 고려 중인 기능이 있다고 했는데, 구체적으로 뭐야?

  - **A**: README에 따르면 key 값을 파라미터로 추가해 캐시 매니징을 통한 자동 업데이트를 고려하고 있습니다. 또한 서버사이드용 함수를 추가해 서버에서 생성한 Atom을 서버->클라이언트에 동기화하는 기능도 고려 중입니다.

- **Q**: key 값을 파라미터로 추가하면 어떤 이점이 있어?

  - **A**: 같은 key를 가진 atom들을 그룹으로 관리할 수 있어, key로 atom을 조회하거나 일괄 업데이트할 수 있습니다. 또한 캐시 전략을 구현할 때 key를 기준으로 무효화할 수 있어 더 유연한 캐시 관리가 가능합니다.

- **Q**: 서버-클라이언트 동기화를 구현하려면 어떻게 해야 할까?
  - **A**: 서버에서 생성한 atom의 초기 상태를 직렬화해 클라이언트로 전달하고, 클라이언트에서 이를 역직렬화해 atom을 복원해야 합니다. 또한 서버에서 상태 변경이 발생하면 클라이언트에 알리는 메커니즘(WebSocket, Server-Sent Events 등)이 필요합니다. 이를 통해 서버와 클라이언트의 상태를 동기화할 수 있습니다.

## 10. 실무 경험 및 문제 해결

### 10.1 개발 과정에서의 문제

- **Q**: 구현 중 가장 어려웠던 부분은?

  - **A**: 파생 atom의 비동기 처리와 구독자 최신화가 가장 어려웠습니다. 특히 Promise를 throw하고 완료 후 재평가하는 과정에서 race condition을 방지하는 것이 까다로웠습니다. 또한 의존성이 동적으로 변경될 때 구독을 정확히 관리하는 것도 복잡했습니다.

- **Q**: 파생 Atom의 비동기 처리에서 어떤 문제를 겪었어?

  - **A**: 여러 비동기 작업이 동시에 실행될 때, 나중에 시작된 Promise가 먼저 완료되면 오래된 결과로 상태가 덮어쓰이는 문제가 있었습니다. 이를 해결하기 위해 `state.suspense === thrown` 비교를 추가해 현재 추적 중인 Promise인지 확인하도록 했습니다.

- **Q**: React 연동 시 발생한 문제들을 어떻게 해결했어?
  - **A**: DESIGN.md에 따르면 메모이제이션 누락으로 구독 콜백이 교체되는 문제가 있었습니다. `useCallback`으로 구독/스냅샷 함수를 메모이제이션해 해결했습니다. 또한 파생 atom이 자신을 의존 atom으로 참조하는 순환 참조 문제도 있었는데, 의존성 추적 시 자기 자신을 제외하도록 해결했습니다.

### 10.2 디버깅

- **Q**: 구독이 제대로 해제되지 않는 문제를 어떻게 디버깅했어?

  - **A**: 구독 해지 함수가 올바르게 반환되는지, 해지 시 Set에서 제거되는지 확인했습니다. 또한 마지막 구독자 해지 시 Set이 null로 설정되는지 확인했습니다. 테스트 코드에서 구독 해지 후 콜백이 호출되지 않는지 검증했습니다.

- **Q**: 무한 루프나 메모리 누수를 어떻게 발견하고 해결했어?
  - **A**: 파생 atom이 자신을 의존하는 경우를 발견하고, 의존성 추적 시 자기 자신을 제외하도록 수정했습니다. 메모리 누수는 구독 해지가 제대로 되는지, WeakMap 캐시가 정리되는지 확인했습니다. 테스트에서 구독 해지 후에도 atom이 정상 작동하는지 검증했습니다.

### 10.3 학습 및 참고

- **Q**: 이 프로젝트를 하면서 새로 배운 게 있어?

  - **A**: `useSyncExternalStore`의 동작 원리와 Concurrent Rendering에서의 안전성, Proxy를 활용한 Suspense 패턴 구현, WeakMap을 활용한 메모리 관리 등을 배웠습니다. 또한 프레임워크 비종속 설계의 장점과 트레이드오프도 이해하게 되었습니다.

- **Q**: Jotai나 Recoil 같은 라이브러리를 참고했어?

  - **A**: Atom 기반 설계는 Jotai에서 영감을 받았지만, 프레임워크 비종속 설계와 API는 독자적으로 설계했습니다. Recoil의 Selector 개념도 참고했지만, 더 단순한 `createDerivedAtom` API로 구현했습니다.

- **Q**: React의 `useSyncExternalStore`를 처음 알게 된 건가?
  - **A**: 네, 이 프로젝트를 통해 처음 알게 되었습니다. 외부 스토어를 React와 연동하는 공식 방법이라는 것을 배웠고, Concurrent Rendering과의 호환성도 이해하게 되었습니다.

## 11. 코드 리뷰 및 개선

### 11.1 코드 품질

- **Q**: 코드 리뷰를 받는다면 어떤 부분을 개선하고 싶어?

  - **A**: 에러 핸들링을 더 강화하고, 타입 안정성을 높이며, 성능 최적화를 더 진행하고 싶습니다. 또한 DevTools 지원이나 더 나은 에러 메시지도 추가하고 싶습니다. 테스트 커버리지도 더 높이고 싶습니다.

- **Q**: 타입 안정성을 더 높이려면 어떻게 해야 할까?

  - **A**: 더 엄격한 타입 가드를 추가하고, 브랜드 타입을 사용해 일반 객체와 atom을 구분할 수 있도록 할 수 있습니다. 또한 조건부 타입을 활용해 더 정확한 타입 추론을 제공할 수 있습니다.

- **Q**: 에러 핸들링을 더 강화하려면?
  - **A**: 사용자 친화적인 에러 메시지를 제공하고, 에러 경계(Error Boundary)와 연동할 수 있는 에러 타입을 정의할 수 있습니다. 또한 개발 모드에서 더 자세한 경고 메시지를 제공할 수 있습니다.

### 11.2 확장성

- **Q**: 이 라이브러리를 더 확장하려면 어떤 기능을 추가해야 할까?

  - **A**: DevTools 지원, 미들웨어/플러그인 시스템, 타임트래블/undo-redo, 서버 사이드 렌더링 지원 등을 추가할 수 있습니다. 또한 더 많은 유틸리티 함수나 헬퍼를 제공할 수 있습니다.

- **Q**: 미들웨어나 플러그인 시스템을 추가하려면 어떻게 설계해야 할까?

  - **A**: `set` 호출 전후에 훅을 실행할 수 있는 미들웨어 API를 추가할 수 있습니다. 예를 들어 `atom.use(middleware)` 같은 API로 미들웨어를 등록하고, `set` 호출 시 미들웨어 체인을 실행하도록 할 수 있습니다.

- **Q**: DevTools를 추가하려면 어떻게 해야 할까?
  - **A**: Redux DevTools Extension과 유사하게, atom의 생성/변경/구독 이벤트를 추적하고 시각화할 수 있습니다. `subscribe`를 래핑해 모든 atom 변경을 기록하고, DevTools API로 전송하면 됩니다.

## 12. 실전 활용

### 12.1 사용 시나리오

- **Q**: 어떤 프로젝트에서 이 라이브러리를 사용하면 좋을까?

  - **A**: 작은 규모의 프로젝트나 프로토타입에서 빠르게 상태 관리를 설정하고 싶을 때 좋습니다. 또한 React와 Vue를 혼합 사용하는 프로젝트에서도 유용합니다. 복잡한 미들웨어나 고급 기능이 필요 없는 경우에 적합합니다.

- **Q**: Redux나 Zustand 대신 이 라이브러리를 선택해야 하는 경우는?

  - **A**: 프레임워크 비종속이 필요하거나, 매우 단순한 API를 원할 때 선택할 수 있습니다. 또한 atom 단위로 세밀한 구독이 필요한 경우에도 유용합니다. 하지만 Redux의 미들웨어나 Zeitgeist 같은 고급 기능이 필요하면 Redux가 더 적합합니다.

- **Q**: 대규모 프로젝트에서 사용할 때 고려해야 할 점은?
  - **A**: atom이 많아지면 관리가 어려워질 수 있으므로, atom을 모듈별로 분리하고 네이밍 컨벤션을 정해야 합니다. 또한 DevTools나 디버깅 도구가 없어 문제 추적이 어려울 수 있습니다. 성능 모니터링도 필요할 수 있습니다.

### 12.2 마이그레이션

- **Q**: 기존 Redux 프로젝트를 이 라이브러리로 마이그레이션하려면?

  - **A**: 점진적으로 마이그레이션할 수 있습니다. 새로운 기능은 이 라이브러리를 사용하고, 기존 Redux 코드는 그대로 유지하면서 점진적으로 교체합니다. Redux의 액션/리듀서를 atom의 `set`으로 변환하고, `useSelector`를 `useAtom`으로 교체합니다.

- **Q**: 점진적 도입이 가능해?

  - **A**: 네, 가능합니다. 프레임워크 비종속이므로 기존 상태 관리 라이브러리와 함께 사용할 수 있습니다. 새로운 컴포넌트나 기능에만 이 라이브러리를 사용하고, 기존 코드는 그대로 유지할 수 있습니다.

- **Q**: React와 Vue를 혼합해서 사용하는 프로젝트에서 이 라이브러리를 사용할 수 있어?
  - **A**: 네, 가능합니다. `@sangtae-js`는 프레임워크 비종속이므로, React 컴포넌트에서는 `@sangtae-react`의 `useAtom`을, Vue 컴포넌트에서는 `@sangtae-vue`의 `useAtom`을 사용하면서 같은 atom을 공유할 수 있습니다.

## 13. 심화 질문

### 13.1 동시성

- **Q**: React 18의 동시성 기능(Concurrent Features)과 어떻게 연동해?

  - **A**: `useSyncExternalStore`가 내부적으로 Concurrent Features와 호환되도록 설계되어 있습니다. 렌더링 중 스냅샷이 변경되면 경고를 발생시켜 일관성을 보장하고, 자동 배치와도 연동되어 효율적으로 동작합니다.

- **Q**: 여러 Atom을 동시에 업데이트할 때 일관성을 어떻게 보장해?

  - **A**: 각 atom은 독립적으로 업데이트되지만, React의 자동 배치로 인해 같은 이벤트 핸들러 내에서 여러 atom을 업데이트하면 한 번의 리렌더로 처리됩니다. 하지만 트랜잭션 같은 원자적 업데이트는 지원하지 않습니다.

- **Q**: `useSyncExternalStore`가 Concurrent Rendering에서 안전한 이유는?
  - **A**: `useSyncExternalStore`는 렌더링 중 스냅샷을 고정하고, 렌더링 완료 후에만 새로운 스냅샷을 적용합니다. 이렇게 하면 Concurrent Rendering 중에 상태가 변경되어도 렌더링 중에는 일관된 스냅샷을 유지할 수 있습니다.

### 13.2 타입 시스템

- **Q**: TypeScript의 고급 타입 기능(제네릭, 조건부 타입 등)을 어떻게 활용했어?

  - **A**: 제네릭을 사용해 `createAtom<T>`, `get<T>`, `set<T>` 등 모든 API에서 타입을 추론하도록 했습니다. 조건부 타입은 사용하지 않았지만, 제네릭 제약이나 유니온 타입을 활용해 타입 안정성을 높였습니다.

- **Q**: 타입 추론이 잘 되는 편이야?

  - **A**: 네, 잘 됩니다. `createAtom(0)`으로 생성하면 자동으로 `Atom<number>`로 추론되고, 이후 `get`, `set`에서도 타입이 자동으로 추론됩니다. `useAtom`도 제네릭 없이 atom 타입을 자동 추론합니다.

- **Q**: `useAtom`에서 제네릭 없이도 타입이 추론되는 이유는?
  - **A**: TypeScript의 타입 추론이 함수 매개변수에서 제네릭을 자동으로 추론하기 때문입니다. `useAtom(atom)`에서 `atom`의 타입이 `Atom<T>`이면, TypeScript가 `T`를 자동으로 추론해 반환 타입도 `[T, (newValue: T) => void]`로 결정됩니다.

### 13.3 번들 크기

- **Q**: 라이브러리의 번들 크기는 어느 정도야?

  - **A**: 정확한 수치는 측정하지 않았지만, 매우 작습니다. 핵심 API만 제공하고 불필요한 의존성이 없어 최소한의 크기를 유지합니다. tsup으로 트리 쉐이킹도 잘 되어 사용하지 않는 코드는 제외됩니다.

- **Q**: 트리 쉐이킹이 잘 되는 편이야?

  - **A**: 네, 잘 됩니다. 각 함수가 독립적으로 export되어 사용하지 않는 함수는 번들에서 제외됩니다. ESM을 사용하면 정적 분석이 가능해 트리 쉐이킹이 더 잘 됩니다.

- **Q**: 번들 크기를 더 줄이려면 어떻게 해야 할까?

  - **A**: 불필요한 코드를 제거하고, 폴리필을 최소화하며, 더 작은 라이브러리로 대체할 수 있는 부분을 찾아야 합니다. 또한 코드 스플리팅이나 동적 import를 활용할 수 있습니다.

- **Q**: "초경량"이라고 했는데, 구체적으로 얼마나 작아? 다른 상태 관리 라이브러리와 비교했을 때?

  - **A**: 정확한 수치는 측정하지 않았지만, Redux나 Zustand보다 훨씬 작을 것으로 예상됩니다. 핵심 기능만 제공하고 불필요한 기능이 없어 최소한의 크기를 유지합니다.

- **Q**: ESM/CJS 모두 지원하는데, 어떻게 번들링했어?
  - **A**: tsup이 자동으로 ESM과 CJS 두 가지 형식으로 번들을 생성합니다. `package.json`의 `exports` 필드로 각 형식을 지정하고, 사용 환경에 맞게 자동으로 선택됩니다.

## 14. 실제 사용 및 API 설계

### 14.1 API 사용 패턴

- **Q**: `$counter`처럼 변수명 앞에 `$`를 붙이는 컨벤션을 사용한 이유는?

  - **A**: README 예제에서 atom임을 명확히 구분하기 위해 `$` 접두사를 사용했습니다. 이는 선택사항이지만, atom과 일반 변수를 구분하기 쉽게 해줍니다. Svelte의 `$` 구독 문법과 유사한 컨벤션입니다.

- **Q**: `useAtom`이 `[value, setValue]` 튜플을 반환하는 이유는? 다른 패턴(객체 반환 등)을 고려했어?

  - **A**: React의 `useState`와 동일한 패턴을 따르기 위해 튜플을 반환했습니다. 이렇게 하면 기존 React 개발자에게 친숙하고, 구조 분해 할당도 간단합니다. 객체 반환(`{ value, setValue }`)도 가능하지만, 튜플이 더 간결하고 널리 사용되는 패턴입니다.

- **Q**: 컴포넌트 밖에서 `get`, `set`, `subscribe`를 직접 사용하는 경우는 언제야?
  - **A**: 로깅, 분석, 상태 동기화 등 컴포넌트와 무관한 사이드 이펙트를 처리할 때 사용합니다. 예를 들어 atom 값 변경을 로그로 남기거나, 다른 시스템과 동기화할 때 유용합니다. 또한 유틸리티 함수나 헬퍼에서도 사용할 수 있습니다.

### 14.2 에러 처리

- **Q**: 비동기 Atom에서 에러가 발생하면 어떻게 처리해?

  - **A**: Promise가 reject되면 `state.status`를 `ERROR`로 설정하고 `state.error`에 에러를 저장합니다. `get` 호출 시 `ERROR` 상태면 에러를 throw해 사용자가 try-catch로 처리할 수 있습니다. `getAsync`를 사용하면 Promise reject가 자동으로 전파됩니다.

- **Q**: 파생 Atom 계산 중 에러가 발생하면 어떻게 해?

  - **A**: `evaluate` 함수의 catch 블록에서 에러를 처리합니다. Promise가 아닌 일반 에러면 `state.status`를 `ERROR`로 설정하고 `emit()`으로 구독자에게 알립니다. `suppressErrors`가 false면 에러를 다시 throw해 상위로 전파합니다. Proxy의 `get` 트랩에서 `ERROR` 상태면 에러를 throw합니다.

- **Q**: 사용자가 잘못된 값을 `set`으로 전달하면 어떻게 처리해?
  - **A**: 현재는 타입 체크만 수행하고 런타임 검증은 하지 않습니다. TypeScript를 사용하면 컴파일 타임에 타입 에러를 발견할 수 있지만, 런타임에서는 `Object.is` 비교만 수행합니다. 필요하다면 개발 모드에서 값 검증을 추가할 수 있습니다.

### 14.3 개발자 경험(DX)

- **Q**: 개발자 경험을 위해 어떤 부분을 신경 썼어?

  - **A**: 간단한 API, 자동 타입 추론, 명확한 에러 메시지(향후 개선 예정), 일관된 패턴(React의 `useState`와 유사) 등을 고려했습니다. 또한 프레임워크 비종속 설계로 다양한 환경에서 사용할 수 있도록 했습니다.

- **Q**: 디버깅을 쉽게 하기 위해 어떤 기능을 제공해?

  - **A**: 현재는 기본적인 기능만 제공하지만, 향후 DevTools 지원이나 로깅 기능을 추가할 계획입니다. 현재는 브라우저 개발자 도구에서 atom 객체를 직접 확인하거나, `subscribe`로 값 변경을 추적할 수 있습니다.

- **Q**: 타입 에러 메시지가 명확한 편이야?
  - **A**: TypeScript의 기본 타입 에러 메시지를 사용하므로, 대부분의 경우 명확합니다. 하지만 더 나은 에러 메시지를 위해 커스텀 타입 가드나 에러 타입을 추가할 수 있습니다.
